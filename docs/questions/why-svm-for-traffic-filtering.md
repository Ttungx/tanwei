# SVM 流量初筛方案选型调研报告

> **调研日期**: 2026-04-07
> **调研问题**: 为什么选择自研 SVM 模型做流量初筛，而不是采用业界已有的轻量级方案？
> **结论**: 业界轻量级方案在"资源约束"和"检测目标"之间存在断层，没有发现同时满足四个核心条件的成熟方案。

---

## 1. 问题背景

Tanwei 边缘智能终端系统采用 **两级检测架构**：

```
流量 → SVM 初筛（过滤正常流量）→ LLM 深度分析（精确识别）
```

其中 `svm-filter-service` 作为第一级，使用自研 SVM 模型进行流量行为异常检测。本报告回答：**为何不采用业界已有的轻量级流量检测方案？**

---

## 2. 核心约束条件

根据 `docs/design-docs/core-beliefs.md`，边缘场景的硬约束如下：

| 约束维度 | Tanwei 要求 |
|----------|-------------|
| **内存** | < 500MB（容器分配 300MB） |
| **延迟** | 微秒级（< 1ms） |
| **部署** | 离线，无云端依赖 |
| **检测目标** | **流量行为异常检测**（非签名匹配、非协议识别） |

任何替代方案必须同时满足上述四个条件。

---

## 3. 业界方案调研

### 3.1 方案清单（按内存占用排序）

#### 方案 A: Go-DPI（轻量 DPI 库）

| 维度 | 数据 |
|------|------|
| 内存占用 | **~20-50MB** |
| 推理延迟 | 微秒级 |
| 检测方式 | 协议识别 |
| 成熟度 | ⭐⭐⭐（社区维护） |
| 开源地址 | github.com/negbie/go-dpi |

**优点**：
- 极轻量，Go 实现
- 支持常见协议识别

**致命缺陷**：
- **只能做协议/应用识别，无法检测恶意行为**
- Go-DPI 告诉你这是 "HTTPS" 流量，但不知道是否是恶意 HTTPS
- 没有"正常/异常"二分类能力
- 社区活跃度较低

**匹配度判定**：❌ 不适用（检测目标不匹配）

---

#### 方案 B: nDPI（NTOP 开源 DPI 库）

| 维度 | 数据 |
|------|------|
| 内存占用 | **~50-100MB** |
| 推理延迟 | 微秒级 |
| 检测方式 | 协议识别 + 应用分类 |
| 成熟度 | ⭐⭐⭐⭐⭐（ntop 核心组件，广泛部署） |
| 开源地址 | github.com/ntop/nDPI |

**优点**：
- 极轻量，纯 C 实现
- 支持 200+ 协议识别
- 被广泛用于流量监控（ntopng、pfSense 等）

**致命缺陷**：
- **只能做协议/应用识别，无法检测恶意行为**
- nDPI 输出的是协议类型（如 "TLS"、"DNS"、"BitTorrent"），不是安全判定
- 无法区分"正常 HTTPS"和"恶意 HTTPS"

**匹配度判定**：❌ 不适用（检测目标不匹配）

---

#### 方案 C: Falco（CNCF 运行时安全）

| 维度 | 数据 |
|------|------|
| 内存占用 | **~150-250MB** |
| 推理延迟 | 毫秒级 |
| 检测方式 | 系统调用规则匹配 |
| 成熟度 | ⭐⭐⭐⭐⭐（CNCF 毕业项目） |
| 开源地址 | github.com/falcosecurity/falco |

**优点**：
- 轻量级，专为容器设计
- eBPF 驱动，内核级检测
- 丰富的规则库（容器逃逸、提权等）

**致命缺陷**：
- **检测目标完全不同**：Falco 检测的是**系统调用异常**（进程行为），不是**网络流量异常**
- Falco 无法分析 Pcap 文件，无法做流量分类
- 适用于"容器运行时安全"，不适用于"流量行为检测"

**匹配度判定**：❌ 不适用（检测目标不匹配）

---

#### 方案 D: Cilium（eBPF 网络安全）

| 维度 | 数据 |
|------|------|
| 内存占用 | **~300-500MB**（Agent 模式） |
| 推理延迟 | 纳秒级（内核级） |
| 检测方式 | 网络策略执行 + 可观测性 |
| 成熟度 | ⭐⭐⭐⭐⭐（CNCF 毕业） |
| 开源地址 | github.com/cilium/cilium |

**优点**：
- 极低延迟，eBPF 内核级
- L3-L7 网络策略执行
- Hubble 可观测性

**致命缺陷**：
- **是策略执行引擎，不是检测引擎**
- Cilium 执行你定义的规则（如"允许端口 443"），不会告诉你"这个流量可疑"
- 没有 ML/异常检测能力
- 需要 K8s 环境，Tanwei 是单机容器部署

**匹配度判定**：❌ 不适用（是执行层不是检测层）

---

#### 方案 E: Snort 3（轻量配置）

| 维度 | 数据 |
|------|------|
| 内存占用 | **~300-500MB**（轻量配置） |
| 推理延迟 | 微秒~毫秒级 |
| 检测方式 | 签名规则匹配 |
| 成熟度 | ⭐⭐⭐⭐⭐（行业标准） |
| 开源地址 | github.com/snort3/snort3 |

**优点**：
- 成熟的签名检测
- 丰富的规则库（Emerging Threats、Talos 等）
- 可配置内存占用

**缺陷**：
- **签名检测对零日攻击无效**：只能检测已知攻击签名
- 需要持续更新规则库（离线环境困难）
- Tanwei 的目标是**行为异常检测**，不是签名匹配
- 勉强能跑，但会挤压 LLM 内存空间（总容器上限 2.3GB）

**匹配度判定**：⚠️ 勉强可用，但检测目标不匹配 + 资源紧张

---

#### 方案 F: Zeek（原 Bro IDS）

| 维度 | 数据 |
|------|------|
| 内存占用 | **~500MB-2GB**（取决于流量） |
| 推理延迟 | 毫秒~秒级 |
| 检测方式 | 协议分析 + 脚本规则 |
| 成熟度 | ⭐⭐⭐⭐⭐（学术/企业广泛使用） |
| 开源地址 | github.com/zeek/zeek |

**优点**：
- 强大的协议分析能力
- 可扩展脚本语言（Zeek Script）
- 被动式，不干扰流量

**缺陷**：
- **内存开销过大**：标准配置需要 1GB+，轻量配置仍需 ~500MB
- 在 Tanwei 的 300MB 内存约束下难以稳定运行
- 规则驱动，需要编写检测脚本，对未知威胁能力有限

**匹配度判定**：❌ 资源约束不满足

---

#### 方案 G: Suricata

| 维度 | 数据 |
|------|------|
| 内存占用 | **~1-4GB**（生产环境推荐 8GB+） |
| 推理延迟 | 毫秒~秒级 |
| 检测方式 | 签名规则匹配 + 协议分析 |
| 成熟度 | ⭐⭐⭐⭐⭐（广泛部署） |
| 开源地址 | github.com/OISF/suricata |

**缺陷**：
- **内存开销远超约束**：最低配置也需 1GB+
- 签名驱动，对零日攻击能力有限
- 在 Tanwei 的 300MB 约束下完全无法运行

**匹配度判定**：❌ 资源约束严重不满足

## 4. 核心发现

### 4.1 业界方案的"检测目标断层"

```
┌─────────────────────────────────────────────────────────────────┐
│                     业界方案分布                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   资源轻量 (<500MB)          │        资源重度 (>500MB)          │
│   ─────────────────          │        ─────────────────          │
│   • nDPI (协议识别)          │        • Zeek (协议分析+规则)     │
│   • Go-DPI (协议识别)        │        • Suricata (签名检测)      │
│   • Falco (系统调用监控)     │        • 商业 NGFW               │
│   • Cilium (策略执行)        │                                   │
│                                                                 │
│   ❌ 无法检测流量行为异常     │        ❌ 资源超出边缘约束         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**没有找到同时满足以下四个条件的成熟方案：**

1. 内存 < 500MB
2. 微秒级延迟
3. 离线部署
4. **流量行为异常检测**（非签名匹配）

### 4.2 检测目标的差异

| 检测目标 | 典型方案 | 与 Tanwei 的匹配度 |
|----------|----------|-------------------|
| 协议识别 | nDPI、Go-DPI | ❌ 无法判断恶意性 |
| 签名匹配 | Snort、Suricata | ⚠️ 零日攻击无效 |
| 系统调用监控 | Falco | ❌ 检测目标不同 |
| 策略执行 | Cilium | ❌ 是执行层不是检测层 |
| **流量行为异常** | **SVM（自研）** | **✅ 完全匹配** |

## 5. SVM 方案对比分析

### 5.1 当前 SVM 方案指标

| 指标 | 数值 |
|------|------|
| 内存占用 | **~300MB** |
| 推理延迟（中位数） | **33.5 μs** |
| 推理延迟（P99） | **203.6 μs** |
| 正常流量过滤率 | 91.4% |
| 异常检测率 | 63.3% |
| 离线部署 | ✅ 完全支持 |

### 5.2 SVM vs 业界方案对比

| 维度 | 业界方案（综合） | SVM（自研） |
|------|------------------|-------------|
| **内存约束** | 勉强或无法满足 | ✅ 完全满足（300MB） |
| **延迟要求** | 部分满足 | ✅ 完全满足（33μs） |
| **离线部署** | 部分依赖规则更新 | ✅ 模型本地 |
| **检测目标** | 协议识别/签名/系统调用 | ✅ 流量行为异常 |
| **可定制性** | 受限于框架设计 | ✅ 完全可控 |
| **零日攻击** | 签名方案无效 | ⚠️ 有一定能力 |

### 5.3 架构设计合理性

Tanwei 的两级检测架构：

```
流量输入
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  SVM 初筛层                                                  │
│  • 内存: 300MB                                               │
│  • 延迟: 33μs                                                │
│  • 功能: 快速过滤 91% 正常流量                                │
│  • 目标: 降低下游 LLM 负载                                   │
└─────────────────────────────────────────────────────────────┘
    │
    ▼ (异常流量)
┌─────────────────────────────────────────────────────────────┐
│  LLM 深度分析层                                              │
│  • 内存: 1GB                                                 │
│  • 延迟: <100ms                                              │
│  • 功能: 精确识别流量语义                                     │
│  • 目标: 弥补 SVM 检测率不足                                  │
└─────────────────────────────────────────────────────────────┘
```

**设计优势**：
- SVM 快速过滤大量正常流量，降低 LLM 负载
- LLM 兜底处理 SVM 漏检的异常，弥补检测率不足
- 两者协同，目标实现 **带宽压降率 >70%**

**如果用业界方案替代 SVM**：
- 用防火墙/nDPI：无法做行为分析，LLM 需要处理所有流量 → **系统过载**
- 用 Snort/Zeek：资源开销过大 → **挤压 LLM 内存空间**

## 6. 结论与建议

### 6.1 调研结论

**SVM 方案不是"重复造轮子"，而是针对边缘场景的合理设计选择。**

业界方案存在"检测目标断层"：
- 轻量方案只能做协议识别/策略执行，无法检测恶意行为
- 能检测恶意行为的方案资源开销过大，不满足边缘约束

SVM 方案在资源约束、延迟要求、离线部署、行为检测四个维度上同时满足 Tanwei 边缘场景需求。

### 6.2 文档化表述建议

在项目文档中可如此解释技术选型：

> SVM 流量初筛层的设计决策基于以下调研：
>
> 1. **业界轻量级方案（nDPI、Go-DPI、Falco、Cilium）**：内存开销低（20-500MB），但检测目标为协议识别、系统调用监控或策略执行，无法进行流量行为异常检测。
>
> 2. **业界流量检测方案（Snort、Suricata、Zeek）**：具备恶意流量检测能力，但资源开销过大（500MB-4GB+），在边缘容器的 300MB 内存约束下难以稳定运行；且依赖签名规则，对零日攻击检测能力有限。
>
> 3. **SVM 方案优势**：在资源约束（300MB）、延迟要求（33μs）、离线部署三个维度完全满足 Tanwei 边缘场景需求；32 维特征向量捕获流量行为模式，配合下游 LLM 深度分析，形成"快速初筛 + 精细分析"的两级检测架构。
>
> 综上，当前没有发现同时满足资源约束和检测目标的成熟开源方案，SVM 方案是合理的技术选择。

### 6.3 潜在改进方向

| 时间尺度 | 改进方向 | 说明 |
|----------|----------|------|
| 短期 | 提升 SVM 异常检测率 | 当前 63% → 目标 75%+ |
| 中期 | 引入轻量规则引擎 | 如 eBPF 规则匹配，补充检测已知攻击 |
| 长期 | 探索替代模型 | Random Forest、Gradient Boosting 等 |
